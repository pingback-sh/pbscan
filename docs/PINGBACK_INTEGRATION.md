# PingBack API v1 integration

## Normal mode

`pbscan` uses the public automation API, not listener-scoped browser dashboard endpoints.

One-time setup:

```bash
pbscan auth --token pba_your_revocable_token --accept-authorized-use
```

Normal scan:

```bash
pbscan urls.txt
```

No listener host, numeric listener ID, dashboard token, callback URL, or `/api/feed.php?t=...` URL is required from the operator.

## API sequence

### 1. Create the listener

```http
POST /api/v1/listeners.php
Authorization: Bearer pba_...
Content-Type: application/json

{"action":"create","label":"pbscan · target.example · pbs-..."}
```

The returned numeric listener ID is saved in `session.json`.

### 2. Register every attempt

Before sending a target request, pbscan calls:

```http
POST /api/v1/injections.php
Authorization: Bearer pba_...
Content-Type: application/json
```

The body includes:

- `listener_id`;
- a private label;
- `vulnerability_type: SSRF`;
- the unmodified target URL;
- exact vector and injection point;
- request method;
- a responsible HTTP request containing `{{PINGBACK_HTTP_PAYLOAD}}` at the selected location;
- scan ID, local attempt ID, and source metadata.

The returned values are stored on the local attempt:

```json
{
  "correlation_id": "inj-a1b2c3",
  "payloads": {
    "http": "https://listener.pingback.sh/cb?cid=inj-a1b2c3",
    "dns": "inj-a1b2c3.listener.pingback.sh",
    "smtp": "inj-a1b2c3@listener.pingback.sh"
  }
}
```

The HTTP payload returned by PingBack is inserted into the selected query parameter, JSON string, form value, or header. pbscan does not manufacture or guess a callback format in API mode.

### 3. Dispatch the target request

Target traffic is controlled independently by:

- `--threads`;
- `--rate`;
- `--timeout`;
- `--retries`;
- redirect and TLS options.

### 4. Poll captured evidence

```http
GET /api/v1/hits.php?listener_id=42&since_id=0&limit=250
Authorization: Bearer pba_...
```

pbscan follows `meta.next_since_id` and `meta.has_more`. Each hit is matched against the attempt's API-provided `correlation_id`. DNS and HTTP events generated by the same injection are retained as separate observations under one finding.

### 5. Resume later

`session.json` stores:

- API base URL, without the token;
- listener ID and host;
- local attempt IDs;
- PingBack correlation IDs and returned payloads;
- last processed hit ID;
- target dispatch results and findings.

The token is loaded from the protected config file or `PINGBACK_API_TOKEN` when `pbscan watch` resumes.

## Token handling

The revocable token is stored only in the local config file with mode `0600`. It is never written to scan output, `session.json`, terminal progress lines, target requests, or GitHub artifacts.

Use a separate token for pbscan so it can be revoked independently.

## API request budget

PingBack documents a default allowance of 600 requests per token per hour. A scan requires approximately:

```text
1 listener request
+ 1 injection request per attempt
+ hit polling requests
```

For that reason, API mode defaults to `--max-attempts 500`. The limit is configurable, but raising it can cause HTTP 429 responses unless the account's allowance is higher.

## Registration failure behavior

Target requests are not dispatched until every injection record has been created successfully. If registration fails partway through, pbscan attempts to delete the newly created listener because no target traffic has yet been sent.

## Legacy compatibility

The former manual mode is still available:

```bash
pbscan urls.txt --legacy --listener https://abc.pingback.sh --feed-url https://example/feed
```

Legacy mode uses local `pba-*` path tokens and generic JSON-feed searching. It should not be used as a replacement for PingBack API v1 when a Pro API token is available.
